USER: What are unit tests and how are they written in Java?
----------------------------------------------------------------

CLAUDE:
Unit Tests
----------
Unit tests verify that individual pieces of code (a single method or class)
behave correctly in isolation. The goal is to test one "unit" of logic at a
time, independent of databases, networks, or other external systems.

Core Concepts:
- Arrange  — set up the inputs and dependencies
- Act      — call the method under test
- Assert   — verify the output matches expectations

Java Unit Testing with JUnit 5
-------------------------------
The standard library is JUnit 5 (often paired with Mockito for mocking dependencies).

Maven dependency:
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.10.0</version>
        <scope>test</scope>
    </dependency>

The class under test:
    public class Calculator {
        public int add(int a, int b) { return a + b; }
        public int divide(int a, int b) {
            if (b == 0) throw new IllegalArgumentException("Cannot divide by zero");
            return a / b;
        }
    }

The test class:
    import org.junit.jupiter.api.*;
    import static org.junit.jupiter.api.Assertions.*;

    class CalculatorTest {

        private Calculator calculator;

        @BeforeEach
        void setUp() {
            calculator = new Calculator();
        }

        @Test
        void add_twoPositiveNumbers_returnsSum() {
            int result = calculator.add(2, 3);
            assertEquals(5, result);
        }

        @Test
        void divide_byZero_throwsException() {
            assertThrows(IllegalArgumentException.class,
                () -> calculator.divide(10, 0));
        }
    }

Key Annotations:
    @Test        — Marks a method as a test case
    @BeforeEach  — Runs before every test (setup)
    @AfterEach   — Runs after every test (teardown)
    @BeforeAll   — Runs once before all tests in the class
    @DisplayName — Human-readable name for the test

Common Assertions:
    assertEquals(expected, actual);
    assertTrue(condition);
    assertFalse(condition);
    assertNull(value);
    assertNotNull(value);
    assertThrows(ExceptionType.class, () -> methodThatThrows());

Mocking with Mockito:
    @Test
    void getUser_callsRepository() {
        UserRepository mockRepo = Mockito.mock(UserRepository.class);
        Mockito.when(mockRepo.findById(1)).thenReturn(new User("Alice"));

        UserService service = new UserService(mockRepo);
        User result = service.getUser(1);

        assertEquals("Alice", result.getName());
        Mockito.verify(mockRepo).findById(1);
    }

Good Test Naming Convention:
    methodName_condition_expectedBehavior
    e.g. divide_byZero_throwsException

USER: How do I know if the unit tests are enough?
----------------------------------------------------------------

CLAUDE:
There's no universal rule, but here are the main things to check:

1. Coverage — are all paths tested?
   Each function should have tests for:
   - Happy path  — normal expected input
   - Edge cases  — boundaries like 0, empty string, single element
   - Error cases — inputs that should throw exceptions

2. Ask yourself for each function:
   - What happens with the smallest/largest valid input?
   - What happens with invalid input?
   - Are there any special cases the code handles explicitly?

Looking at our tests, here's an honest assessment of gaps:

    add            — Could test integer overflow
    isPrime        — Could test 0 explicitly
    reverse        — Good
    factorial      — Could test large numbers
    isPalindrome   — Could test empty string
    fibonacciUpTo  — Could test n=1
    charFrequency  — Could test spaces/special chars
    isAnagram      — Could test same string twice
    average        — Could test negative numbers
    filterEvens    — Could test all-even list
    mostCommonWord — Could test a tie

3. The practical standard for a class like this:
   Our 33 tests cover all happy paths, all documented edge cases, and all
   throws cases — which is solid for a unit testing exercise. In a production
   codebase you'd also run a code coverage tool (like JaCoCo) to see exactly
   which lines are hit, aiming for 80%+ coverage.

USER: Why are there 3 actionable tasks and not 11 as the amount of functions?
----------------------------------------------------------------

CLAUDE:
The 3 Gradle tasks and the 11 functions are completely unrelated things.
They refer to different concepts:

The 3 Gradle tasks are build steps:
    1. compileJava     — compiles App.java
    2. compileTestJava — compiles AppTest.java
    3. test            — runs all the tests

The 11 functions are what's being tested inside step 3.

Gradle doesn't count functions — it counts the build pipeline steps needed
to go from source code to test results. The actual test count (33) happens
entirely within the test task.
